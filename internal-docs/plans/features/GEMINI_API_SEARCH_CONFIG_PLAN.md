# Gemini API Search Configuration Feature Plan

## Feature Goal (Reformulated)

**Primary Goal:** Enable users to describe their file search needs in natural language, and have the Gemini API return a complete search configuration (as JSON) that can be applied to all search options in the application.

**Key Requirements:**
1. **Natural Language Input**: User provides a description (e.g., "find all C++ source files modified this week that are larger than 1MB")
2. **Complete Configuration**: Gemini API returns JSON containing ALL search parameters (inputs, filters, options)
3. **Unambiguous Format**: JSON schema ensures no ambiguity in parsing and applying the configuration
4. **Direct Application**: The JSON configuration can be directly applied to `GuiState` to configure the search

## Current Search Configuration Options

Based on codebase analysis, here are all the search parameters that need to be configurable:

### 1. Search Input Fields (`SearchInputField`)
- **`filenameInput`**: Filename pattern (supports glob `*`, `?`, regex `re:`, path pattern `pp:`)
- **`extensionInput`**: File extensions (comma-separated, e.g., "txt,doc,pdf")
- **`pathInput`**: Path pattern (supports glob, regex, path pattern syntax)

### 2. Search Options (Booleans)
- **`foldersOnly`**: Only search for directories/folders
- **`caseSensitive`**: Case-sensitive vs case-insensitive search
- **`instantSearch`**: Search as you type (with debounce) - *UI behavior, may not need in JSON*
- **`autoRefresh`**: Re-run search when index changes - *UI behavior, may not need in JSON*

### 3. Time Filters (`TimeFilter` enum)
- `None` - No time filter
- `Today` - Files modified today (since midnight)
- `ThisWeek` - Files modified this week (since Monday)
- `ThisMonth` - Files modified this month (since 1st of month)
- `ThisYear` - Current calendar year
- `Older` - More than 1 year ago

### 4. Size Filters (`SizeFilter` enum)
- `None` - No size filter
- `Empty` - 0 bytes
- `Tiny` - < 1 KB
- `Small` - 1 KB - 100 KB
- `Medium` - 100 KB - 10 MB
- `Large` - 10 MB - 100 MB
- `Huge` - 100 MB - 1 GB
- `Massive` - > 1 GB

## Final JSON Schema

**Current Format (Path-Only):**

The prompt now generates only the `path` field, which matches the FULL file path (directory + filename + extension). All matching criteria (extensions, folders, exclusions) are expressed in the path pattern.

```json
{
  "version": "1.0",
  "search_config": {
    "path": "pp:src/**/*.cpp",     // Optional: path pattern matching FULL paths (directory + filename + extension)
                                    // Prefer pp: prefix for simple patterns, use rs: for regex (alternation, exclusions)
    "folders_only": false,         // Optional: default false
    "case_sensitive": false,       // Optional: default false
    "time_filter": "ThisWeek",      // Optional: "None", "Today", "ThisWeek", "ThisMonth", "ThisYear", "Older"
    "size_filter": "Large"          // Optional: "None", "Empty", "Tiny", "Small", "Medium", "Large", "Huge", "Massive"
  }
}
```

**Backward Compatibility:**

The parsing code still supports `filename` and `extensions` fields for backward compatibility with old responses, but the new prompt does not generate them.

### Schema Rules:
- All fields in `search_config` are **optional** (partial configs allowed)
- **Defaults**: Empty strings for inputs, `false` for booleans, `"None"` for filters
- **Path patterns**: Match FULL file paths (directory + filename + extension)
  - Use `pp:` prefix for simple patterns (single extensions, folders, wildcards)
  - Use `rs:` prefix for regex patterns (multiple extensions with alternation, exclusions with lookahead)
  - PathPattern (`pp:`) does NOT support alternation `(a|b)` or negative lookahead `(?!...)`
- **Enum values**: Must match exactly (case-sensitive)

## Decisions Made

### 1. **Scope of Configuration**
✅ **Decision**: Only search parameters (no UI behavior settings like `instantSearch`, `autoRefresh`)

### 2. **Input Field Types**
✅ **Decision**: Use prefixes in the value (`"pp:src/**/*.cpp"`, `"re:^test.*"`, `"rs:pattern"`)
- **Preference**: Prefer `pp:` (path pattern) over other types when possible to simplify
- Matches current UI behavior

### 3. **Extension Handling** (UPDATED)
✅ **Decision**: Extensions are now included directly in the path pattern
- **Single extension**: `pp:**/*.cpp` (path pattern)
- **Multiple extensions**: `rs:.*\.(cpp|hpp|cxx|cc)$` (regex with alternation - PathPattern doesn't support alternation)
- **Extension + exclusion**: `rs:^(?!.*[/]thirdparty[/]).*\.cpp$` (regex with negative lookahead)
- The old `extensions` field is still supported for backward compatibility but is not generated by the new prompt

### 4. **Partial Configuration**
✅ **Decision**: Allow partial configurations with defaults
- Missing fields use defaults (empty strings, `false` for booleans, `None` for filters)
- Allows Gemini to return only relevant fields

### 5. **Natural Language Prompt**
✅ **Decision**: No context (don't include current search state)
- Keep prompts simple and focused
- Each request is independent

### 6. **Error Handling**
- Validate JSON schema before applying
- Show user-friendly error messages
- Invalid enum values default to `None`

## Implementation Requirements

### 1. JSON Schema Definition
- Create a JSON schema file (e.g., `SearchConfigSchema.json`) for validation
- Use a JSON schema library (e.g., `nlohmann/json` with schema validation)

### 2. Gemini API Prompt Engineering
- Update `BuildGeminiPrompt()` to request JSON output instead of just path pattern
- Include JSON schema in the prompt for structured output
- Add examples showing natural language → JSON configuration

### 3. JSON Parsing & Validation
- Add `ParseSearchConfigJson()` function to `GeminiApiUtils`
- Validate JSON against schema
- Map JSON values to `GuiState` fields

### 4. GuiState Integration
- Add method `GuiState::ApplySearchConfig(const SearchConfigJson& config)`
- Handle enum conversions (string → `TimeFilter`, `SizeFilter`)
- Handle input field types (detect glob/regex/path pattern prefixes)

### 5. UI Integration
- Update existing Gemini input field to show "Generate Search Config" button
- On success, apply configuration to `GuiState` and trigger search
- Show preview of configuration before applying (optional)

## Alternative Approaches

### Alternative 1: Structured Text Output (Not JSON)
- **Pros**: Simpler parsing, more human-readable
- **Cons**: Less unambiguous, harder to validate
- **Verdict**: JSON is better for unambiguous parsing

### Alternative 2: Incremental Configuration
- Instead of full config, return only changed fields
- **Pros**: More flexible, can modify existing searches
- **Cons**: More complex, requires current state context
- **Verdict**: Start with full config, add incremental later if needed

### Alternative 3: Multiple API Calls
- One call for path pattern, another for filters, etc.
- **Pros**: Simpler individual prompts
- **Cons**: Multiple API calls, slower, more complex
- **Verdict**: Single call with JSON is better

### Alternative 4: Template-Based Approach
- Use a template system where Gemini fills in values
- **Pros**: Guaranteed structure
- **Cons**: Less flexible, harder to extend
- **Verdict**: JSON schema validation is better

## Recommended Approach

1. **Use JSON with explicit schema** - Unambiguous, easy to validate
2. **Start with complete configuration** - Simpler to implement and use
3. **Support both string and array formats** for extensions (for flexibility)
4. **Use prefix-based input types** (matches current UI behavior)
5. **Validate JSON schema** before applying configuration
6. **Show user-friendly errors** if validation fails
