# UX Review - 2026-01-18

## Executive Summary
- **UX Health Score**: 7/10
- **Top 3 Usability Issues**: Unintuitive default layout, lack of discoverability for advanced features, and missing keyboard navigation.
- **Accessibility Gaps**: The application is not accessible via keyboard and lacks clear focus indicators.
- **Estimated Remediation Effort**: 10-15 hours

## Findings

### High
1.  **Unintuitive Default Layout**
    *   **Location**: `src/ui/SearchInputs.cpp`
    *   **Issue Type**: Information Architecture & Layout
    *   **User Impact**: By default, the "Manual Search" section is collapsed, while the "AI-Assisted Search" section is expanded. This is counter-intuitive for a power-user tool where manual search is the primary function. New users may be confused and not realize that manual search options are available.
    *   **Current Behavior**: The UI prioritizes the AI search feature, hiding the more commonly used manual search tools.
    *   **Recommended Solution**: Change the default behavior to have the "Manual Search" section expanded and the "AI-Assisted Search" section collapsed.
    *   **Severity**: High
    *   **Effort**: S (< 2hrs)

2.  **Poor Discoverability of Advanced Features**
    *   **Location**: `src/ui/SearchInputs.cpp`
    *   **Issue Type**: Discoverability & Learnability
    *   **User Impact**: Advanced search syntax (e.g., regex, wildcards, prefixes like `vs:`) is supported but not documented anywhere in the UI. A user would have to read the documentation to know these features exist.
    *   **Current Behavior**: The search input field is a simple text box with no hints or help buttons.
    *   **Recommended Solution**: Add a small help icon (?) next to the search input field that opens a popup or a separate window (`SearchHelpWindow.cpp`) explaining the available search syntax with examples.
    *   **Severity**: High
    *   **Effort**: M (2-8hrs)

### Medium
1.  **Lack of Keyboard Navigation**
    *   **Location**: Entire UI.
    *   **Issue Type**: Accessibility & Inclusivity
    *   **User Impact**: Power users who rely on the keyboard for navigation will be forced to use the mouse, which is slower and less efficient. Users with motor disabilities may not be able to use the application at all.
    *   **Current Behavior**: Based on the ImGui code, there is no explicit handling of keyboard focus or navigation between UI elements.
    *   **Recommended Solution**:
        -   Ensure all interactive elements can be reached and activated using the Tab and Enter keys.
        -   Implement keyboard shortcuts for common actions (e.g., `Ctrl+F` to focus the search bar, `Ctrl+R` to recrawl).
    *   **Severity**: Medium
    *   **Effort**: L (> 8hrs)

2.  **No Visual Indication of Active Filters**
    *   **Location**: `src/ui/FilterPanel.cpp`
    *   **Issue Type**: Feedback & Status Communication
    *   **User Impact**: When a size or time filter is active, there is no persistent visual indicator. A user might forget that a filter is active and be confused as to why their search is returning no results.
    *   **Current Behavior**: The filter controls are in a separate panel, but once the panel is closed, there is no visible reminder that a filter is applied.
    *   **Recommended Solution**: Add a small "Filters Active" badge or icon to the status bar when any filter is enabled. Clicking this badge could open the filter panel.
    *   **Severity**: Medium
    *   **Effort**: M (2-8hrs)

3.  **Missing "Clear Search" Button**
    *   **Location**: `src/ui/SearchInputs.cpp`
    *   **Issue Type**: User Workflow & Task Flow
    *   **User Impact**: To start a new search, the user must manually delete the text from all the search input fields. This is inefficient.
    *   **Current Behavior**: There is no single button to clear all search inputs.
    *   **Recommended Solution**: Add a "Clear" button next to the "Search" button that resets all search and filter fields to their default state.
    *   **Severity**: Medium
    *   **Effort**: S (< 2hrs)

4.  **Inconsistent Terminology ("Crawl" vs. "Index")**
    *   **Location**: Across the UI.
    *   **Issue Type**: Consistency & Standards
    *   **User Impact**: The UI uses the terms "crawl" and "index" interchangeably (e.g., "Crawl Folder" button, but "Indexing..." status). This can be confusing to users.
    *   **Current Behavior**: Inconsistent terminology.
    *   **Recommended Solution**: Standardize on a single term. "Indexing" is more descriptive of the process and should be used consistently.
    *   **Severity**: Medium
    *   **Effort**: S (< 2hrs)

### Low
1.  **No Confirmation for Destructive Actions**
    *   **Location**: `src/ui/SettingsWindow.cpp`
    *   **Issue Type**: Error Prevention & Recovery
    *   **User Impact**: The "Clear Index" button is a destructive action that does not ask for confirmation. A user could accidentally click it and lose their entire index.
    *   **Current Behavior**: Clicking "Clear Index" immediately deletes the index.
    *   **Recommended Solution**: Show a confirmation dialog before performing any destructive actions.
    *   **Severity**: Low
    *   **Effort**: S (< 2hrs)

2.  **Lack of Perceived Performance During Indexing**
    *   **Location**: `src/ui/StatusBar.cpp`
    *   **Issue Type**: Performance Perception
    *   **User Impact**: When the index is being built, the UI shows the number of entries processed, but it doesn't provide a percentage or an estimated time to completion. This can make the process feel slow and unpredictable.
    *   **Current Behavior**: The status bar shows "Indexing... (12345 entries)".
    *   **Recommended Solution**: If possible, estimate the total number of files to be indexed and show a progress bar with a percentage.
    *   **Severity**: Low
    *   **Effort**: M (2-8hrs)

## Summary Requirements
- **UX Health Score**: 7/10. The application is functional and powerful, but it has several usability issues that detract from the user experience, especially for new users. The core functionality is solid, but the UI lacks polish and attention to detail.
- **Top 3 Usability Issues**:
    1.  The default layout, which hides the primary manual search functionality.
    2.  The lack of discoverability for advanced search features.
    3.  The complete absence of keyboard navigation.
- **Accessibility Gaps**: The application is not accessible. Keyboard navigation and focus indicators are the most significant gaps.
- **Quick Wins**:
    -   Change the default collapsed state of the search sections.
    -   Add a "Clear Search" button.
    -   Add a confirmation dialog for the "Clear Index" button.
- **Design System Recommendations**: The application would benefit from a more consistent design system, especially for spacing, alignment, and terminology.
- **User Journey Assessment**: The primary user journey for a power user (manual search) is hindered by the default layout. Once found, the search functionality is powerful, but the workflow could be made more efficient with the addition of a clear button and better filter indicators.
